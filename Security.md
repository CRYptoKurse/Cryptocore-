# Политика безопасности

## Поддерживаемые версии

| Версия | Поддерживается |
|--------|----------------|
| 1.x.x  | ✅             |
| < 1.0  | ❌             |

## Сообщение об уязвимости

**НЕ сообщайте об уязвимостях в публичных issues.**

Сообщите по email: security@example.com

Ответ в течение 48 часов.

### Что включить
- Тип уязвимости
- Пути к файлам
- Местоположение кода
- Шаги воспроизведения
- Код эксплойта (если есть)
- Возможное воздействие

### Шифрование
Для чувствительных отчетов используйте PGP:
-----BEGIN PGP PUBLIC KEY BLOCK-----
[PGP ключ]
-----END PGP PUBLIC KEY BLOCK-----

text

## Вопросы безопасности

### Реализация криптографии
- Алгоритмы соответствуют стандартам
- Используются тестовые векторы
- Постоянные по времени операции
- Криптографический ГСЧ

### Управление ключами
- Ключи не в коде
- Очистка памяти после использования
- Правильные рабочие факторы для KDF

### Проверка входных данных
- Проверка длины и формата
- Проверка границ буферов
- Проверка типов

### Обработка ошибок
- Без утечки информации
- Очистка ресурсов
- Безопасные неудачи

## Рекомендации для пользователей

### Ключи
- Достаточная энтропия
- Безопасное хранение
- Регулярная смена
- Не в коде приложений

### Пароли
- Сильные и уникальные
- Менеджеры паролей
- Соответствующие итерации PBKDF2

### Операции
- Проверка подлинности
- Использование AAD
- Обновление ПО
- Мониторинг уведомлений

## Обновления безопасности

- **Критические:** 24 часа
- **Высокие:** 72 часа
- **Средние:** Следующий релиз
- **Низкие:** Следующий мажорный релиз

## Политика раскрытия

1. **Приватное раскрытие:** Сообщитель → сопровождающие
2. **Подтверждение:** Получение подтверждено
3. **Расследование:** Подтверждение уязвимости
4. **Патч:** Разработка исправления
5. **Тестирование:** Внутреннее тестирование
6. **Релиз:** Выпуск исправленной версии
7. **Публичное раскрытие:** Через 90 дней или с патчем

## Аудит безопасности

- Ежегодный полный аудит
- Автоматическое сканирование с каждым релизом
- Мониторинг уязвимостей зависимостей
